# PBI-001: Source control integration (clone on enqueue, push on gate pass)

## Directive
Implement the source control lifecycle described in
`specs/source-control-integration/spec.md`: the orchestrator clones or
pulls the target repo before enqueuing a PBI, and the worker pushes the
agent's commits to a remote branch after all gates pass.

This PBI modifies the factory's orchestrator and worker. It must be
gated behind a `SOURCE_CONTROL_ENABLED` environment variable that
defaults to false, so existing local-directory behaviour is completely
unaffected when the flag is off.

## Context Pointer
Spec: specs/source-control-integration/spec.md

## Scope
You may modify:
- `factory/orchestrator/src/` — add clone/pull logic on enqueue
- `factory/worker/src/` — add push logic after gate pass
- `factory/.env.example` — document the new variables
- `factory/db/init.sql` — add `push_status` and `remote_branch` columns
  to the `run` table if needed for the API response

Do not add:
- Any UI for configuring git credentials
- Any merge-to-default-branch logic — branch push only, per the spec

## Depends On
PBI-002

## Verification Pointer
See specs/source-control-integration/spec.md#contract — all five
Definition of Done items, with particular attention to the flag-off
regression guardrail (existing behaviour must be byte-for-byte identical
when `SOURCE_CONTROL_ENABLED=false`).

## Refinement Rule
If the git operations (clone, pull, push) need a library rather than
shelling out to the `git` CLI, prefer `simple-git` (well-maintained Node
wrapper) over a raw `child_process` shell-out — fewer edge cases around
auth token injection in URLs. If `simple-git` is chosen, note the
dependency addition in the commit message.
